# 业务数据权限控制 - 产品规格说明

## Summary

在现有用户体系和组织架构基础上，实现业务数据级别的权限控制框架。系统支持多种数据权限控制策略，各功能模块的开发者根据业务需求在后端代码中为所在模块指定适用的权限策略。当前支持的策略包括：**纯组织架构控制**和**人员-数据绑定 + 组织架构混合控制**。不同用户登录后，系统根据其所属模块的权限策略自动过滤可查看的数据范围。上级可查看下级拥有的数据权限范围内的数据。支持管理员配置用户的数据权限（针对需要绑定关系的模块），并记录操作日志审计。

## Problem

当前系统仅有功能菜单级别的权限控制（角色-菜单/功能关联），缺乏业务数据级别的权限控制。所有有功能权限的用户都能看到相同的数据集，无法满足以下业务需求：

- 不同功能模块的数据敏感度和管控要求不同：有的模块只需按组织隔离（如部门数据），有的模块需要精确到个人负责范围（如客户数据）
- 管理者需要查看下属负责的数据汇总
- 数据安全要求敏感数据仅对授权人员可见

## Goals

- 提供一套可扩展的数据权限控制框架，支持至少两种控制策略：
  - **策略一：纯组织架构控制**——按用户所属组织及其下级组织过滤数据
  - **策略二：人员-数据绑定 + 组织架构控制**——在组织层级基础上，增加人员与业务数据的精细绑定关系
- 各功能模块开发者可在后端代码中为本模块指定使用哪种权限策略
- 查询接口根据当前登录用户及所在模块的权限策略自动过滤数据，无需前端额外处理
- 针对使用策略二的模块，提供管理员配置界面，支持分配/回收用户的数据权限
- 所有权限变更操作记录审计日志

## Non-goals

- 本阶段仅实现**查看（Read）**权限控制，新增/修改/删除操作的权限控制后续迭代实现
- 不涉及字段级别的数据脱敏或加密
- 不涉及数据导出时的权限过滤（后续迭代）
- 不改变现有的功能菜单权限体系
- 不提供运行时动态切换策略的能力（策略由开发时确定，非配置项）

## Behavior

### 1. 数据权限控制策略

系统提供两种数据权限控制策略，各功能模块的开发者在开发该模块时，根据业务需求在后端代码中**显式指定**本模块使用哪种策略。策略一旦确定后，该模块所有受控查询接口均按此策略执行过滤。

#### 策略一：纯组织架构控制（ORG）

1. 适用场景：模块的数据天然按组织归属（如部门公告、组织配置等），无需精确到个人负责范围。
2. 用户的数据访问范围由其所在组织及其下级组织决定：
   - 当前用户可查看**本人所属组织**及**所有下级组织**范围内的数据。
3. 过滤依据：业务数据表中的 **`org_id`** 字段（标识数据归属哪个组织）。
4. 数据访问范围计算：`org_id IN (当前用户所属组织ID + 所有下级组织ID列表)`。
5. 此策略**不需要**人员-数据绑定关系，也不需要 `t_data_permission` 表。

#### 策略二：人员-数据绑定 + 组织架构混合控制（OWNER_ORG）

6. 适用场景：模块的数据需要精确到个人负责范围（如客户数据、产品数据），同时管理者需要汇总查看下属负责的数据。
7. 每个业务数据记录可绑定一个或多个**负责人（owner）**，负责人为系统用户。
8. 用户的数据访问范围由以下规则共同决定：
   - **本人数据**：当前用户作为负责人的所有业务数据记录。
   - **下级数据**：组织架构中当前用户的所有下级用户（直接或间接下级）作为负责人的业务数据记录。
   - 数据范围为「本人数据 ∪ 所有下级数据」的并集。
9. 过滤依据：通过 `t_data_permission` 绑定表确定负责人关系，再结合组织架构计算上下级范围。
10. 此策略**依赖** `t_data_permission` 绑定表和 `t_sys_org` 组织架构表。

#### 策略指定方式

11. 各模块开发者通过后端代码中的声明式注解（如 `@DataPermission(strategy = DataPermissionStrategy.ORG)` 或 `@DataPermission(strategy = DataPermissionStrategy.OWNER_ORG)`）为模块的查询接口指定权限策略。
12. 若某查询接口未标注任何权限策略注解，默认行为为**不进行数据权限过滤**（向后兼容已有接口）。
13. 策略在代码编译时确定，运行时不可动态切换。若需变更策略，需修改代码后重新部署。

#### 策略扩展性

14. 框架应预留扩展点，使未来新增权限策略（如「仅本人数据」「自定义规则」等）时，只需新增策略枚举值和对应的 SQL 拼接逻辑，不影响已有策略的行为。

### 2. 组织架构层级关系

15. 组织架构基于 `t_sys_org` 表的 `parentId` 字段形成树形层级结构。
16. 用户归属组织通过 `t_sys_user.orgId` 确定。
17. **上下级判定逻辑**：用户 A 是用户 B 的上级，当且仅当：
    - 用户 A 所在组织是用户 B 所在组织的祖先节点（包括父节点、祖父节点等），或
    - 用户 A 与用户 B 属于同一组织，且用户 A 在系统中被指定为管理者（后续可通过管理字段扩展）。
18. **初始版本简化**：上下级关系仅通过组织架构的父子关系判定——同组织内的用户视为平级，互不可见对方无绑定关系的数据。
19. **下级组织/用户列表计算**：
    - 对于策略一（ORG）：需要计算当前用户所属组织的**所有下级组织 ID 列表**（递归遍历 `parentId` 树）。
    - 对于策略二（OWNER_ORG）：需要计算当前用户的**所有下级用户 ID 列表**（先找下级组织，再取这些组织下的所有用户）。

### 3. 查询接口自动过滤

20. 所有标注了数据权限注解的业务数据查询接口（列表查询、分页查询、详情查询），必须在服务层根据当前登录用户及该接口声明的权限策略自动添加过滤条件。
21. 当前登录用户身份从 `UserContext`（已存在的用户上下文）中获取。

#### 策略一（ORG）的过滤行为

22. **列表/分页查询过滤**：SQL 查询自动追加 WHERE 条件，仅返回当前用户所属组织及其下级组织范围内的数据记录。过滤条件为：`org_id IN (当前用户所属组织ID + 所有下级组织ID列表)`。
23. **详情查询过滤**：查询单条数据时，若该记录的 `org_id` 不在当前用户的组织权限范围内，返回「无权限访问该数据」的错误提示，HTTP 状态码 403。

#### 策略二（OWNER_ORG）的过滤行为

24. **列表/分页查询过滤**：SQL 查询自动追加 WHERE 条件，仅返回当前用户及其下级用户作为负责人的数据记录。过滤条件通过关联 `t_data_permission` 表实现：业务数据记录存在于绑定表中，且 `user_id IN (当前用户ID + 所有下级用户ID列表)`。
25. **详情查询过滤**：查询单条数据时，若该记录不在当前用户及其下级的负责范围内，返回「无权限访问该数据」的错误提示，HTTP 状态码 403。

#### 通用行为

26. **未登录状态**：若请求未携带有效会话信息（无法获取当前用户），返回「未登录或会话已过期」错误提示，HTTP 状态码 401。此规则对两种策略均适用。

### 4. 数据权限绑定关系（仅策略二）

27. 新增一张「人员-数据权限绑定表」（`t_data_permission`），**仅用于策略二（OWNER_ORG）**，记录用户与业务数据实体的所有权关系：
    - `id`: 主键
    - `user_id`: 用户 ID（负责人）
    - `data_type`: 业务数据类型标识（如 `customer`、`product`），预留扩展多数据类型
    - `data_id`: 业务数据记录的主键 ID
    - `createUserId`, `createdTime`, `updateUserId`, `updateTime`：通用审计字段
28. 一条业务数据记录可有多个负责人（多对多关系），任一负责人及其上级均可查看该数据。
29. 绑定关系的变更是管理员操作，普通用户不能自行修改自己的数据权限绑定。
30. 使用策略一（ORG）的模块**不使用此表**，其数据权限完全由业务数据表的 `org_id` 字段决定。

### 5. 管理员配置权限

31. 具有「数据权限管理」功能权限的管理员（超级管理员默认拥有），可以执行以下操作：
    - **分配数据权限**（仅策略二模块）：将某条业务数据记录分配给指定用户（在 `t_data_permission` 中建立绑定关系）
    - **回收数据权限**（仅策略二模块）：解除某用户与某条业务数据记录的绑定关系
    - **批量分配**（仅策略二模块）：将多条数据记录批量分配给同一用户
    - **查看权限分布**（仅策略二模块）：查看某条数据有哪些负责人，或某用户负责哪些数据
32. 对于策略一（ORG）的模块，管理员通过**调整用户的组织归属**（修改 `t_sys_user.orgId`）来间接控制用户的数据访问范围，无需额外的权限配置界面。
33. 分配数据权限时，目标用户必须为有效状态（`status = 1`）的正常用户；若用户已禁用或不存在，操作失败并给出明确提示。
34. 回收数据权限时，若该用户是此条数据的唯一负责人，系统给出警告提示「该数据将没有任何负责人，确认继续？」，管理员确认后方可执行。
35. 数据权限变更后，对应用户的下次查询请求立即生效（无需重新登录）。此规则对两种策略均适用：
    - 策略一：用户组织调整后立即生效
    - 策略二：绑定关系变更后立即生效

### 6. 操作日志审计

36. 以下数据权限相关操作必须记录到操作日志表（`t_sys_oper_log` 或新建专用审计表）：
    - **策略二模块**：数据权限分配、数据权限回收、批量分配（包含操作人、目标用户、数据类型、数据 ID、操作时间）
    - **策略一模块**：用户组织归属变更（包含操作人、目标用户、原组织、新组织、操作时间）
37. 日志记录内容至少包含：操作类型、操作人 ID 和姓名、目标对象信息、操作时间、操作结果（成功/失败及原因）。
38. 操作日志仅允许新增记录，不允许修改或删除（ append-only ）。

### 7. 边界情况与异常处理

39. **新用户无数据权限**：
    - 策略一（ORG）：用户刚创建且已分配组织后，可正常查看其所属组织及下级组织的数据。若用户未分配组织（`orgId` 为空），查询返回空数据集。
    - 策略二（OWNER_ORG）：用户刚创建且未被分配任何数据权限绑定时，查询受控接口返回空数据集（非报错）。
40. **用户被禁用**：用户被禁用（`status != 1`）后，其会话失效，再次请求返回 401。其作为负责人的数据绑定关系保留不解绑（策略二），上级仍可查看这些数据；其组织归属不变（策略一）。
41. **组织调整**：用户从一个组织调转到另一个组织后：
    - 策略一：数据权限范围立即按新组织层级重新计算。
    - 策略二：上下级关系随之变化，数据权限范围自动重新计算。
42. **数据被删除**：
    - 策略一：业务数据记录被物理删除即可，无需额外清理。
    - 策略二：业务数据记录被物理删除时，关联的 `t_data_permission` 绑定记录应级联删除（数据库外键约束或应用层清理）。
43. **循环依赖防护**：组织架构数据出现循环引用（A 的 parent 是 B，B 的 parent 是 A）时，上下级计算应检测环并终止遍历，避免无限循环，同时记录告警日志。此规则对两种策略均适用。
44. **并发冲突处理**：
    - 策略一：两个管理员同时调整同一用户的组织归属时，以最后提交的结果为准。
    - 策略二：两个管理员同时对同一数据分配权限时，以最后提交的结果为准。不使用乐观锁或悲观锁（最终一致性即可）。

### 8. API 行为规范

45. 数据权限过滤对调用方**透明**：前端调用查询接口时无需传递任何权限相关参数，后端根据当前会话用户及接口声明的策略自动处理。
46. 受控接口的响应格式不变，仅在数据集层面做了过滤。分页接口的 `totalCount` 也应为过滤后的数量（而非全量数量），确保前端分页正确。此规则对两种策略均适用。
47. 若某个接口不需要数据权限过滤（如公开数据、字典数据等），需在接口或 Service 层显式标注跳过权限过滤（通过自定义注解 `@DataPermissionIgnore` 或不标注 `@DataPermission` 注解）。

### 9. 性能要求

48. **下级列表缓存**：
    - 策略一：下级组织 ID 列表的查询结果必须缓存（建议使用 Redis），缓存过期时间不超过 5 分钟，组织架构变更时主动清除缓存。
    - 策略二：下级用户 ID 列表的查询结果必须缓存（建议使用 Redis），缓存过期时间不超过 5 分钟，组织架构变更时主动清除缓存。
49. 权限过滤 SQL 应使用 `IN (...)` 子查询或 JOIN，确保在数据量增长时查询性能不严重退化。此规则对两种策略均适用。
50. 对于百万级以上的业务数据表：
    - 策略一：需确保业务数据表的 `org_id` 字段有数据库索引。
    - 策略二：需确保 `t_data_permission` 表的 `(data_type, data_id)` 和 `(user_id)` 字段有联合索引。

## Open questions

1. **同组织内用户的可见性（策略一）**：初始版本同组织内平级用户之间是否完全不可见其他组织的数据？还是需要增加「同级组织互不可见，但可看父级汇总」的选项？
2. **超级管理员的数据范围**：超级管理员是否默认可以看到所有数据（不受任何权限策略过滤限制），还是也需要遵循各模块声明的权限策略？
3. **业务数据类型的扩展方式（策略二）**：当前以 `data_type` 字符串区分数据类型，未来新增数据类型时是否需要改动代码，还是可以通过配置动态扩展？
4. **策略选择指导**：是否需要在文档或代码注释中提供明确的策略选择决策树，帮助开发者为新模块判断应使用哪种策略？
