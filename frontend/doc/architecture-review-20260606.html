Problem

Eight components each re-implement identical fetch/paginate/modal/crud logic. Every bug fix or pattern change must be replicated across all eight. The interface of each module is as complex as its implementation — maximum shallowness.

Solution

Extract a useCrudTable<T>() hook that owns the entire CRUD lifecycle. Each page component becomes a thin adapter that only supplies column definitions and API endpoints.

Wins

  • • Locality: CRUD bugs live in one module
  • • Leverage: 8 call sites share one interface
  • • Delete ~2,500 lines of copy-pasted code
  • • Pagination behavior changes once, not 8 times
  • • Error handling is consistent by construction
  • • New CRUD pages take ~40 lines instead of ~400

Problem

Every component calls fetch() directly with hand-rolled error handling, JSON parsing, and data.success checks. Changing the auth header format or error response shape touches 20+ files.

Solution

Create an apiClient module that wraps fetch(), injects credentials, unwraps the standard { success, data, errMessage } envelope, and throws typed errors. Two adapters: real HTTP for production, in-memory for tests.

Wins

  • • Seam: HTTP is swappable at one point
  • • Locality: Auth token injection lives in one place
  • • Error handling is consistent by construction
  • • Components test without a running backend
  • • Response envelope change = 1 file, not 20+

Problem

App.tsx mixes four responsibilities: authentication state machine, permission/menu data fetching, layout orchestration, and route definition. Any change to auth flow requires editing the routing/layout component.

Solution

Extract an AuthProvider that owns auth state, session checking, login/logout, and permission loading. App.tsx shrinks to route definitions inside a layout shell.

Wins

  • • Locality: Auth logic isolated from routing
  • • Depth: AuthProvider has high leverage behind small interface
  • • Token refresh / SSO can be added without touching routes
  • • App.tsx drops from 207 to ~60 lines

Four audit columns are copy-pasted across 8+ table components with minor variations in width. A base entity type with createUserId, createdTime, updateUserId, updateTime, createUserName, updateUserName is re-defined locally in every component.

  • • Type safety: all entities share base fields
  • • Eliminates subtle width/format inconsistencies

Standardize error handling through the API seam

Speculative local-substitutable
Pattern A — RoleManagement, NoticeTable, DictTable, FileTable:
  message.error(data.errMessage || fallback) via App.useApp()
Pattern B — Dashboard, LogOper:
  setError(state) + conditional Result UI rendering
Pattern C — ProcessTemplateTable:
  setNotification({ message, type, key }) custom Notification component
Pattern D — ConfigTable, CacheTable, MonitoringTable:
  No error handling at all (mock/static data)
  All patterns also include: console.error() scattered throughout

Problem

Four different error-handling strategies across the codebase. Some show toast notifications, some set error state for UI rendering, some use a custom Notification component, some silently ignore errors. No consistent user experience for failures.

Solution

The API client seam (Candidate 2) should own error translation: network errors become one toast, server errors become another, validation errors get surfaced near the form field. Components stop choosing how to handle errors — they receive standardized results.

Wins

  • • Locality: Error UX defined in one place
  • • Users see consistent error feedback everywhere
  • • console.error() removed from production paths
Note: This candidate is largely subsumed by Candidate 2 (API client seam). Implementing the API client naturally forces error handling standardization. Listed separately because the current inconsistency is itself a maintenance hazard.
Top Recommendation

Candidate 1: Extract a deep CRUD table module

This candidate delivers the highest leverage-to-effort ratio. Eight components spanning 4,000+ lines share an identical fetch-paginate-modal-crud pattern. Extracting a useCrudTable<T>() hook consolidates the most repeated code in the codebase into one testable, deep module.

It also creates natural pressure toward Candidate 2 (API client seam) since the hook needs a clean data-access layer, and toward Candidate 5 (shared types/columns) since the hook's generic parameter demands common interfaces.
