候选 #1

前端缺少统一 API 客户端层

Strong
涉及文件
· src/components/* — 19 个组件，64 处 fetch
· src/App.tsx — 3 处独立 fetch（auth/permissions/menus）
· vite.config.ts — 仅代理配置，无请求拦截器
问题诊断
  • 零复用：每个组件独立编写 fetch → response.json() → data.success 判断 的完整流程
  • 错误处理不一致：有的用 message.error()，有的静默失败，有的 console.error
  • 无类型安全：所有响应都是 any 类型，无接口契约
  • 无统一凭证管理：credentials: 'include' 在每次调用中重复书写
  • 变更成本高：如需添加 token 刷新、请求重试、全局 loading，需修改全部 19 个组件
解决方案

引入统一的 API 客户端模块 src/lib/api.ts，封装通用请求模式：

// src/lib/api.ts — 统一 API 客户端
class ApiClient {
  private baseUrl = '/api';

  async get<T>(path: string): Promise<T> { ... }
  async post<T>(path: string, body?: any): Promise<T> { ... }
  async del<T>(path: string): Promise<T> { ... }

  // 统一错误处理、credentials、超时、类型转换
}

export const api = new ApiClient();

收益：

  • 64 处 fetch → 统一为 api.get()/api.post() 调用
  • 错误处理集中在一处，行为一致
  • 添加 TypeScript 泛型响应类型，编译期捕获接口不匹配
  • 未来可扩展：token 刷新、请求取消、重试策略
Before — 每个组件重复样板代码
// OrgUserManagement.tsx (同样模式在19个组件中重复)
const fetchData = async () => {
  setLoading(true);
  try {
    const res = await fetch('/api/orgs/page', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ pageIndex, pageSize })
    });
    const data = await res.json();
    if (data.success) { ... }
  } catch (e) { ... }
  finally { setLoading(false); }
};
After — 声明式调用
// OrgUserManagement.tsx — 使用统一客户端
const { data, loading, error } = useQuery({
  queryKey: ['orgUsers', orgId, page, size],
  queryFn: () => api.post<PageResult<OrgUser>>(
    '/orgs/page', { pageIndex: page, pageSize: size }
  )
});
深度与杠杆率（Depth & Leverage）：
这是一个高杠杆率的改进。API 客户端的 interface 很小（get/post/del），但背后承载了错误处理、认证、类型安全、缓存等大量行为。 局部性极佳——所有 HTTP 相关的变更只需改一个模块。删除测试通过：如果删除 api.ts，复杂性会扩散到所有 19 个调用者中。
候选 #2

PermissionServiceImpl — 上帝类（God Class）

Strong
涉及文件
· PermissionServiceImpl.java — 553 行，承担 6 种职责
· RolePermissionController.java — 权限分配入口
· AuthController.java — 认证入口（依赖权限查询）
问题诊断：6 种职责混在一个类中
① 权限分配
assignRolePermissions()
事务性写入 + resolveMenuIds
② 菜单查询
getAccessibleMenus()
getRoleMenus()
③ 权限校验
getUserPermissions()
hasPermission() × 2
④ 缓存管理
ConcurrentHashMap
TTL 5分钟 + invalidateCache
⑤ 快照构建
buildPermissionSnapshot()
buildLogRemark() + diff
⑥ 操作日志
savePermissionLog()
JSON 序列化变更对比
  • 违反单一职责原则：修改缓存策略需要理解日志格式，修改日志需要理解权限模型
  • 内部 DTO 泛滥：定义了 7 个私有内部类/记录（CachedPermissions, FunctionDTO, MenuPermissionDTO, SnapshotDTO, ChangesDTO, ChangeDetailDTO, LogRemarkDTO），全部是权限快照相关的
  • 难以测试：无法单独测试"权限校验"逻辑而不触发缓存和数据库操作
解决方案：按职责拆分为深模块
PermissionAssignService
assignRolePermissions() + resolveMenuIds() + savePermissionLog()
→ 分配、快照、日志归为一组（写操作聚合）
PermissionQueryService
getUserPermissions() + getRolePermissions() + getRolesByUserId() + hasPermission() + cache
→ 读操作 + 缓存（读操作聚合）
MenuAccessService
getAccessibleMenus() + getRoleMenus() + addParentMenuIds() + collectParentIds()
→ 菜单可见性判断（领域概念独立）
PermissionSnapshotBuilder
buildPermissionSnapshot() + buildLogRemark() + buildChangeDetails() + 所有内部 DTO
→ 快照构建是纯函数逻辑，可脱离 Spring 容器独立单元测试
Before — 单一 God Class
After — 职责分离的深模块
深度与杠杆率（Depth & Leverage）：
拆分后每个新模块都有清晰的 interface（seam）。SnapshotBuilder 是纯逻辑，可以脱离 Spring 容器进行单元测试。PermissionQueryService 可以独立替换缓存实现（从 ConcurrentHashMap 切换到 Redis）。

删除测试通过：如果删除 SnapshotBuilder，快照构建逻辑不会消失——它会回到 AssignService 中变得更难测试。如果删除 QueryService，缓存和权限查询逻辑会分散到各调用者。两者都在 earn their keep。
候选 #3

Table 组件大量样板代码重复

Worth exploring
涉及文件
· LogOper.tsx / LogLogin.tsx / LogError.tsx — 结构几乎相同
· NoticeTable.tsx / FileTable.tsx / DictTable.tsx — CRUD 模式一致
· OnlineUsersTable.tsx / OrgUserManagement.tsx — 分页+搜索相同
问题诊断
  • 每个 Table 组件都重复定义：currentPage/pageSize/totalCount 三个状态 + fetchData 函数 + 标准分页配置 + try/catch/finally 模式
  • 工作区规范已识别此问题：frontend-pagination-spec.md 定义了完整的分页模板，说明团队意识到一致性需求，但靠文档约束而非代码约束
  • 30 处 setLoading(true) 分布在 19 个文件中——同样的加载状态管理模式被复制了 30 次
  • 变更风险：修改分页行为需同步修改 11+ 个组件
解决方案

抽取通用的 usePaginatedQuery Hook 或 PaginatedTable 容器组件：

// 方案 A: 自定义 Hook（推荐，侵入性最小）
function usePaginatedQuery<T>(
  fetchFn: (page: number, size: number) => Promise<PageResult<T>>
) {
  const [page, setPage] = useState(1);
  const [size, setSize] = useState(20);
  const { data, loading, error, refetch } = useQuery(...);
  const pagination = { /* 标准分页配置 */ };
  return { data, loading, pagination, page, refetch };
}

使用效果：

// LogOper.tsx — 从 ~200行 减少到 ~80行
const { data, loading, pagination } = usePaginatedQuery(
  (p, s) => api.get(`/api/oper-logs?page=${p}&size=${s}`)
);
深度与杠杆率：
这个改进的 interface 非常小（一个 fetchFn 参数），但消除了每个 Table 组件中约 40-60 行的重复代码。局部性好：分页逻辑变更只改一处。
注意：这属于"浅层重构"——它减少的是机械性重复，不改变业务逻辑结构。但如果配合候选 #1（统一 API 客户端）一起做，效果会叠加。
候选 #4

COLA 执行器模式使用不一致

Worth exploring
涉及文件
· RoleServiceImpl.java — ✅ 使用完整 COLA 执行器模式（8 个 Executor）
· PermissionServiceImpl.java — ❌ 直接在 Service 中实现全部逻辑
· AuthServiceImpl.java — ❌ 直接在 Service 中实现
· WorkflowServiceImpl.java — ⚠️ 部分使用执行器
· StatsServiceImpl.java — ❌ 直接在 Service 中实现
问题诊断
  • 两种风格并存：Role 模块严格遵循 COLA 四层链（Controller→Service→CmdExe/QryExe→Converter→Mapper），但 Permission/Auth/Stats 模块跳过了 Executor 层
  • 已有 70+ 个 Executor 文件证明团队认可该模式价值，但未全面推行
  • 认知负担：开发者定位代码前需先判断"这个模块用的是哪种模式"
现状对比
✅ Role 模块 — 标准 COLA 四层调用链
Controller → RoleService(I) → RoleSaveCmdExe → RoleConvertor → Mapper
❌ Permission 模块 — 跳过 Executor 层
Controller → PermissionService(I) → ??? 直接操作 6 个 Mapper ???
建议
  • 新模块强制使用 COLA 执行器模式
  • Permission 模块结合候选 #2 一起重构（拆分时自然形成 Executor）
  • 简单查询模块（如 Stats）保持现状——不需要为 2 行查询创建 Executor
深度与杠杆率：
这是架构一致性问题而非功能性 bug。不一致的模式增加了新人上手成本和维护时的上下文切换成本。但在 Permission 这种复杂度高的模块上补齐 Executor 层，同时解决候选 #2 的 God Class 问题，是一石二鸟的做法。
候选 #5

前后端无共享类型契约

Speculative
涉及范围
· 前端 19 个组件中的 interface 定义（各自手写）
· 后端 project-api 中的 DTO 类
· RolePermissionController 接口不匹配问题（本次实际 bug）
问题诊断
  • 前端接口类型手写且分散：每个组件自行定义 interface User { id: number; username: string; ... }，与后端 DTO 字段逐步偏离
  • 接口变更无编译期保护：后端重命名字段或新增必填项，前端无法感知，直到运行时报错
  • 本次 bug 的根因：前端假设返回 {functionId}[] 对象数组，后端实际返回 Long[] 数字数组——如果有共享类型，编译期就能发现
解决方案选项
方案 A: 共享 types 包（轻量）
在 shared/types/ 中定义接口，前后端各复制一份或用符号链接。成本低，但需人工同步。
方案 B: OpenAPI/Swagger 自动生成（推荐）
后端加 Swagger 注解，前端用 openapi-typescript 自动生成 API 类型和请求函数。一次配置，自动同步。
方案 C: Monorepo 共享包
用 Turborepo/Nx 将 types 抽取为共享 package。最规范，但对当前项目规模可能过重。
深度与杠杆率：
这是一个长期投资类型的改进。短期内看不到明显收益，但随着项目增长（更多 API、更多人协作），类型契约的价值呈非线性增长。标记为 Speculative 是因为当前项目规模下，手动同步的类型定义尚可维持。
首要推荐

优先实施：Candidate #1 + #2 组合拳

🎯

理由：Candidate #1（统一 API 客户端）和 Candidate #2（拆分 PermissionServiceImpl）有最强的协同效应：

  1. API 客户端建立后，前端调错接口的问题会在类型层面被拦截（类似本次 /api/role-permissions 的 bug）
  2. PermissionServiceImpl 拆分后，后端接口边界更清晰，前端可以更准确地对应到每个服务的能力
  3. 两项改进分别作用于前后端，可以并行开发，互不阻塞
  4. 两者完成后，Candidate #3（Table 去重）和 #5（类型契约）的实施难度大幅降低
预估影响范围：前端 19 个组件 + 后端 1 个 Service 类 | 风险评估：低（渐进式重构，不影响现有功能）| 测试策略：API 客户端可并行编写；Permission 拆分需先补充测试覆盖
