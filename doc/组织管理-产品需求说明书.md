## 1. 功能概述

### 1.1 产品定位

组织管理模块是 Zenith 系统的核心基础模块，提供企业级组织架构管理能力，支持：

- ✅ **组织架构管理**：树形结构的组织/部门管理
- ✅ **用户管理**：组织下用户的增删改查、状态管理
- ✅ **角色分配**：支持多角色的灵活分配与调整
- ✅ **权限控制**：基于 RBAC（基于角色的访问控制）模型

### 1.2 核心价值

### 2.1 组织架构管理

#### 2.1.1 功能描述

提供树形结构的组织/部门管理，支持无限层级嵌套。

#### 2.1.2 核心功能

|------|------|---------|
| **查看组织树** | 以树形结构展示所有组织 | 自动展开/折叠 |
| **搜索组织** | 关键词模糊搜索组织名称 | 输入框实时过滤 |
| **新增顶级组织** | 创建根级组织 | 点击"新建"按钮 |

| **编辑组织** | 修改组织名称等信息 | 右键菜单 → "编辑" |
| **删除组织** | 删除空组织（无成员） | 右键菜单 → "删除" |

| **刷新组织** | 重新加载最新数据 | 点击刷新按钮 |

#### 2.1.3 数据库设计

```sql
CREATE TABLE public.t_sys_org (
    id bigserial NOT NULL,                    -- 主键ID，自增序列
    parent_id int8 DEFAULT 0 NULL,            -- 父级组织ID，0表示顶级组织
    "name" varchar(100) NOT NULL,             -- 组织/部门名称

    created_time timestamp DEFAULT CURRENT_TIMESTAMP NULL,  -- 创建时间
    create_user_id int8 NULL,                -- 创建人ID
    update_user_id int8 NULL,                -- 最后更新人ID
    update_time timestamp DEFAULT CURRENT_TIMESTAMP NULL,  -- 最后更新时间
    CONSTRAINT t_sys_org_pkey PRIMARY KEY (id)
);
```

### 2.3 角色分配功能 ⭐核心功能

#### 2.3.1 功能背景

**历史问题**：
- ❌ 角色列表硬编码在前端代码中（`['ADMIN', 'EDITOR', 'USER']`）
- ❌ 只能设置单个角色，无法真正增减
- ❌ 无法显示数据库中的真实角色
- ❌ 无效（禁用）的角色也会出现在可选列表中

**改进目标**：
- ✅ 动态从数据库加载有效角色
- ✅ 支持多角色自由增减
- ✅ 实时同步角色管理页面的数据
- ✅ 只显示启用的有效角色

```
GET /api/user-roles?userId=7

{
```

---

##### 接口 3：更新用户角色（全量替换）

```
POST /api/user-roles

Content-Type: application/json

请求体：
{
  "userId": 7,           // 用户ID (Long, 必填)
  "roleIds": [4, 1]      // 角色ID列表 (List<Long>, 必填)
}

响应示例：
{
  "success": true,
  "errCode": null,
  "errMessage": null
}
```

**处理逻辑**：
1. 删除该用户在 `t_sys_user_role` 表中的所有旧记录
2. 批量插入新的角色关联记录
3. 整个操作在事务中执行，保证原子性

#### 2.3.5 技术实现要点

##### 数据模型变更

**之前（单角色）**：
```
t_sys_user.role = "ROLE_ADMIN"  (单个字符串)
```

**现在（多角色）**：
```
t_sys_user_role 表:
┌────┬─────────┬─────────┐
│ id │ user_id │ role_id │
├────┼─────────┼─────────┤
│  1 │   7     │   3     │  ← 用户7拥有角色3
│  2 │   7     │   4     │  ← 用户7拥有角色4
└────┴─────────┴─────────┘
```

**关键决策**：统一使用 `roleId`（数字ID）而非 `roleCode`（字符串编码）

**原因**：
- ✅ roleId 是主键，唯一且稳定
- ✅ 避免编码不一致导致的匹配失败
- ✅ 后端处理更简单，无需查表转换
- ✅ 性能更优，减少一次数据库查询

##### 后端服务层

**PermissionServiceImpl.java** 核心方法：

```java
/**
 * 获取用户的角色ID列表
 */
@Override
public List<String> getUserRoles(Long userId) {
    // 1. 查询 t_sys_user_role 表
    List<UserRoleDO> userRoles = userRoleMapper.selectList(
        new LambdaQueryWrapper<UserRoleDO>()
            .eq(UserRoleDO::getUserId, userId)
    );
    
    // 2. 提取 roleId 并转为字符串返回
    return userRoles.stream()
        .map(ur -> ur.getRoleId().toString())
        .collect(Collectors.toList());
}

/**
 * 更新用户角色（全量替换）
 */
@Override
@Transactional(rollbackFor = Exception.class)
public void updateUserRoles(Long userId, List<Long> roleIds) {
    // 1. 删除该用户所有旧角色关联
    LambdaQueryWrapper<UserRoleDO> deleteWrapper = new LambdaQueryWrapper<>();
    deleteWrapper.eq(UserRoleDO::getUserId, userId);
    userRoleMapper.delete(deleteWrapper);

    // 2. 直接使用 roleId 批量插入新记录
    if (roleIds != null && !roleIds.isEmpty()) {
        for (Long roleId : roleIds) {
            UserRoleDO userRole = new UserRoleDO();
            userRole.setUserId(userId);
            userRole.setRoleId(roleId);  // 直接使用 roleId
            userRole.setCreatedTime(LocalDateTime.now());
            userRoleMapper.insert(userRole);
        }
    }
}
```

## 8. 已知限制与后续优化

### 8.1 当前限制

| 限制项 | 说明 | 建议 |
|--------|------|------|
| **无批量分配** | 只能逐个用户分配角色 | 后续可增加批量操作 |
| **无操作日志** | 未记录角色变更历史 | 建议增加审计日志 |
| **无权限控制** | 所有登录用户都可分配 | 建议限制只有管理员可操作 |
| **无并发锁** | 高并发可能冲突 | 可考虑乐观锁 |

### 8.2 后续优化方向

- [ ] **批量角色分配**：支持多选用户同时分配相同角色
- [ ] **角色继承**：子组织自动继承父组织的默认角色
- [ ] **角色有效期**：支持临时角色（如实习生角色，到期自动失效）
- [ ] **角色审批流程**：重要角色变更需上级审批
- [ ] **导入导出**：支持 Excel 批量导入用户和角色
