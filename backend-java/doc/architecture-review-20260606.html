Candidate 1: Workflow Task Command Consolidation

Strong

The highest-leverage refactoring opportunity in the codebase

Files Involved

  • TaskTerminateCmdExe.java (51 lines)

  • TaskTransferTaskCmdExe.java (28 lines)
  • TaskAssignApproverCmdExe.java (28 lines)
Duplication Metrics
Task validation pattern 6 copies
Status update + record creation 4 copies

Problem

Six task command executors share an identical 15-line validation sequence: load task → null check → PENDING status check → assignee ownership check. After validation, 4 of them also share the same status-update + approval-record pattern. The node-progression logic (find next/prev node, advance or return the process) is duplicated between TaskApproveCmdExe.processAfterApprove() and TaskRejectCmdExe.execute().

Deletion test result: Deleting any single executor and inlining its unique 3-5 lines would concentrate complexity into a shared module. These are shallow modules pretending to be deep.

Before / After

Before — Scattered Duplication
A TaskApproveCmdExe: validate + approve + findNextNode + advanceOrComplete
R TaskRejectCmdExe: validate + reject + terminateCountersigns + findPrevNode + return
T TaskTerminateCmdExe: validate + terminate + completeInstance
CS TaskCountersignCmdExe: validate + countersign + createChildTasks
TR TaskTransferTaskCmdExe: validate + reassign
AS TaskAssignApproverCmdExe: validate + reassign
Each box = independent module with duplicated validation + status-update code
After — Deepened Module
TaskOperationTemplate (new seam)
validateTask(taskId, userId) → validated TaskDO
recordAction(task, actionType, opinion, userId)
advanceNode(instance, currentOrder) → next or complete
returnNode(instance, currentOrder) → prev or returned
A Approve: template.validate() + setApproved() + template.record() + template.advanceNode()
R Reject: template.validate() + setRejected() + template.record() + template.returnNode()
T Terminate: template.validate() + setTerminated() + completeInstance()
CS/TR/AS Each: only their unique 3-5 lines remain
One deep module with high leverage; callers are thin adapters

Solution

Extract a TaskOperationHelper or WorkflowTaskSeam module that encapsulates:

  • Task validation — load + null-check + status-check + ownership-check, returns validated TaskDO
  • Action recording — set status/opinion/actionType/time + persist + create ApprovalRecord via DomainService
  • Node progression — find next/prev node by order, advance currentNodeOrder or set terminal status

Each command executor becomes a thin adapter calling 2-3 methods on the shared module with its unique parameters. The interface shrinks from N full method bodies to N short orchestration sequences.

Benefits — Locality

  • ✓ Task validation rule change = edit 1 place (not 6)
  • ✓ Status transition logic = auditable in one module
  • ✓ Node progression bugs = isolated to single file
  • ✓ New task actions = add thin adapter only

Benefits — Testability

  • ✓ Test validation logic once, cover all 6 commands
  • ✓ Test node progression as state machine independently
  • ✓ Mock the seam in command executor tests
  • ✓ Current 0 tests on any of these 6 files

Candidate 2: Inconsistent Exception Handling in Workflow Executors

Worth Exploring

Two exception styles coexist without clear boundary

Style A — BizException (Correct)
  • ✓ TaskApproveCmdExe
  • ✓ TaskRejectCmdExe
  • ✓ TaskTerminateCmdExe
  • ✓ TaskCountersignCmdExe
  • ✓ TaskTransferTaskCmdExe
  • ✓ TaskForceEndCmdExe
throw new BizException("TASK_NOT_FOUND", "任务不存在");
Style B — RuntimeException (Incorrect)
  • ✗ WorkflowSubmitCmdExe
  • ✗ WorkflowCancelCmdExe
  • ✗ WorkflowResubmitCmdExe
  • ✗ WorkflowRevokeCmdExe
  • ✗ WorkflowCreateTaskCmdExe
  • ✗ WorkflowResolveApproversQryExe
throw new RuntimeException("流程实例不存在");

Problem

Two distinct exception styles exist in the Executor layer with no architectural reason for the split. Style A uses structured BizException(errorCode, message) which integrates with the global exception handler for consistent error responses. Style B uses raw RuntimeException(chineseString) which bypasses structured error codes, prevents i18n, and may produce inconsistent HTTP responses.

The split happens to follow entity boundaries: Task executors use BizException, while ProcessInstance/Template executors use RuntimeException. This is coincidental, not intentional — there's no ADR documenting this choice.

Solution

Standardize all executors to use BizException with proper error codes. Define a centralized error code constants class for the workflow domain (e.g., WorkflowErrorCode.INSTANCE_NOT_FOUND, WorkflowErrorCode.NODE_NOT_FOUND). This is a mechanical refactor with high value: every workflow error becomes traceable, loggable, and frontend-handleable via error codes.

Candidate 3: Workflow Instance Operation Overlap

Worth Exploring

Submit and Resubmit are nearly identical flows

Problem

// WorkflowSubmitCmdExe.execute()
instance = selectById(id);
check(instance != null);
check(status == DRAFT || RETURNED);

nodes = selectByTemplateId(templateId);
firstNode = nodes.find(order == 1);

instance.status = IN_PROGRESS;
instance.currentNodeOrder = 1;
updateById(instance);

createTasksForNode(id, firstNode.id);
createApprovalRecord(..., SUBMIT);
// WorkflowResubmitCmdExe.execute()
instance = selectById(id);
check(instance != null);
check(status == RETURNED);
check(initiatorId == currentUserId);

if (formData != null)
  instance.formData = formData;

instance.status = IN_PROGRESS;
instance.currentNodeOrder = 1;
updateById(instance);

nodes = selectByTemplateId(templateId);
firstNode = nodes.find(order == 1);
if (firstNode != null)
  createTasksForNode(id, firstNode.id);

createApprovalRecord(..., RESUBMIT);

Both executors perform the same core operation: validate instance → set IN_PROGRESS → find first node → create tasks → record action. The differences are: (1) Resubmit checks initiator ownership and optionally updates formData, (2) Resubmit uses RESUBMIT action type vs SUBMIT. That's 2 conditional branches, not two separate modules.

Solution

Extract a WorkflowLaunchExe or add a launch(ProcessInstanceId, ActionType, formData?, requireInitiatorCheck?) method to a shared workflow operation module. Submit and Resubmit become one-liner delegations with different parameter values. This also sets up the seam for future operations that need to "launch" a workflow from any state.

Benefits

  • ✓ Launch logic (status transition + task creation) defined once
  • ✓ Future "restart from any node" operation plugs into same seam
  • ✓ Reduces 2 executors (~110 combined lines) to 1 shared + 2 thin callers (~30 lines each)

Problem

The workflow engine — the most complex subsystem with state machine behavior, concurrent task handling, countersigning, and multi-node progression — has zero tests. Any bug in task approval, rejection, or node progression propagates directly to production.

Recommended Test Priority

  2. TaskApproveCmdExe — test happy path + countersign-wait + final-node-approval + concurrent-reject
  3. TaskRejectCmdExe — test reject-at-node-1 (returns) vs reject-at-node-N (returns-to-prev)
  4. WorkflowResolveApproversQryExe — test ROLE/USER/SUPERIOR resolver branches
  5. WorkflowSubmitCmdExe — test DRAFT submit + RETURNED resubmit + empty-template guard

Candidate 5: Missing Domain Glossary (CONTEXT.md)

Speculative

No canonical definitions for core domain concepts

Problem

The project has no CONTEXT.md and no ADRs in docs/adr/. Core domain concepts like ProcessInstance, Task, NodeTemplate, ApprovalRecord, Countersign, ActionType have no canonical definitions. When naming new modules or resolving debates about where certain logic belongs, there is no reference document. This makes the codebase harder for AI tools and new developers to navigate.

Solution

Create CONTEXT.md defining the workflow domain glossary. Key terms to capture:

ProcessInstance
A single workflow execution, created from a template
Task
An actionable item assigned to a user at a specific node
NodeTemplate
A step definition within a ProcessTemplate (approve/sign/countersign)
ApprovalRecord
Immutable audit trail entry per user action on a task
Countersign
Node type requiring ALL assignees to approve before advancing
ActionType
Enum: SUBMIT/APPROVE/REJECT/COUNTERSIGN/TERMINATE/CANCEL/REVOKE/RESUBMIT

Why This Matters

  • ✓ AI coding assistants use CONTEXT.md vocabulary for consistent naming
  • ✓ New developers understand domain boundaries without reading all code
  • ✓ ADRs prevent re-litigating settled decisions (like COLA adoption, Executor pattern)
  • ✓ Candidate 1's renamed module gets a canonical name from the glossary
