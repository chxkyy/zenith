Problem

MenuServiceImpl is a 291-line god class that bypasses the Executor seam entirely. Tree algorithms (descendant detection, depth calculation, sibling reordering) live in the wrong layer. Zero locality — business logic scattered across a module whose interface is as wide as its implementation.

Solution

Extract 7 Executor modules from MenuServiceImpl. Move tree operations into a dedicated MenuTreeCmdExe. Service becomes a thin orchestration layer matching the pattern already established by UserServiceImpl and RoleServiceImpl.

Locality: tree bugs concentrate in one Executor Leverage: 7 focused interfaces vs 1 shallow facade Testability: each Executor unit-testable in isolation Deletion test: removing Service would not re-expose complexity

Problem

8 Service implementations directly inject Mappers and manipulate DO objects, bypassing the Executor seam entirely. Meanwhile 9 other services (User, Role, Dict, Org, Task, Workflow, etc.) correctly delegate to Executors. After — Single Utility

Add one method to PageResponseUtils:
public static <D, T> PageInfo<T> convert(
  PageInfo<D> source,
  Function<List<D>, List<T>> converter) {
  PageInfo<T> result = new PageInfo<>();
  result.setTotal(source.getTotal());
  result.setPageNum(source.getPageNum());
  result.setPageSize(source.getPageSize());
  result.setPages(source.getPages());
  result.setList(converter.apply(source.getList()));
  return result;
}

// Caller becomes:
return PageResponseUtils.convert(
  pageInfo, convertor::toDTOList);
✓ 1 definition, N call sites

Problem

The same 6-line PageInfo manual field-copy pattern is copy-pasted across 8 Service implementations (and some Executors). Any change to pagination semantics requires editing 8+ locations. Fails the deletion test: deleting any one copy doesn't eliminate the pattern.

Solution

Add a generic PageInfoUtils.convert(PageInfo, Function) method that encapsulates the DO-to-DTO page conversion. Each call site collapses to one line. The existing PageResponseUtils.of() handles the COLA response wrapping; this handles the intermediate layer.

Locality: pagination logic in one place Leverage: eliminates ~48 lines of duplication Safety: single point of change for pagination bugs

Problem

ProcessTemplateServiceImpl contains 30 lines of private DTO conversion methods (convertToDTO, convertNodeToDTO) that mix domain mapping concerns with service orchestration. Every other entity uses MapStruct-based Convertors — ProcessTemplate is the exception. Also directly operates on two Mappers without an Executor seam.

Solution

Create ProcessTemplateConvertor and NodeTemplateConvertor interfaces using MapStruct (matching the existing pattern for User, Role, Dict, Task etc.). Extract Executors for CRUD and query operations. The timestamp conversion logic in convertToDTO belongs in the Convertor, not the Service.

Locality: mapping rules in Convertor, not Service Consistency: matches all other entities' patterns Testability: Convertor testable independently

Top Recommendation

Candidate #1: Deepen MenuServiceImpl into Executor Layer

MenuServiceImpl is the worst offender by every metric: 291 lines (largest Service), contains complex tree algorithms (isDescendant, getDepth, getMaxChildDepth, reorderSiblings) that are pure business logic living in the wrong layer, and directly manipulates MenuDO through MenuMapper. It represents the highest-leverage refactoring target because:

  • ➤ It establishes the migration pattern for the remaining 8 shallow services
  • ➤ The tree operations are non-trivial code that benefits most from isolation and testing
  • ➤ The existing UserServiceImpl / RoleServiceImpl prove the target architecture works — just replicate it
  • ➤ Once done, Candidates #2 (batch migration) becomes a mechanical, lower-risk effort
