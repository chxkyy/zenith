Candidates Identified

This review analyzed the Zenith backend Java codebase following COLA (Clean Object-Oriented and Layered Architecture) principles. The analysis identified 5 deepening opportunities ranging from eliminating shallow pass-through layers to consolidating scattered business logic into cohesive modules. The overarching theme: the architecture has the right shape (Controller → Service → Executor → Mapper) but many modules are shallow — their interfaces are nearly as complex as their implementations, offering little leverage or locality.

Candidate #1: Service Layer as Pure Pass-Through (Shallow Modules)

Strong

Files Involved

UserServiceImpl.javaMenuServiceImpl.javaRoleServiceImpl.java DictServiceImpl.javaTaskServiceImpl.javaFunctionServiceImpl.java OrgServiceImpl.javaNoticeServiceImpl.javaWorkflowServiceImpl.java WorkflowDomainServiceImpl.javaAuthServiceImpl.java + 7 more ServiceImpl classes

Problem

All 18 ServiceImpl classes are pure pass-through wrappers. Every method is a single-line delegation to an Executor. Zero business logic, zero orchestration, zero value-add.

The deletion test fails catastrophically: if you deleted any of these ServiceImpl classes, complexity would not reappear across callers — Controllers could call Executors directly with no loss of behavior.

Dead parameter pattern detected: Methods like save(UserAddCmd cmd, Long currentUserId), delete(Long id, Long currentUserId), update(UserUpdateCmd cmd, Long currentUserId) accept currentUserId but never forward it to the Executor. The parameter is swallowed. See UserServiceImpl.java:39-51, MenuServiceImpl.java:40-67, and most other services.

Solution

Option A — Flatten for CRUD modules: For simple CRUD entities (User, Menu, Role, Dict, Org, Function, Notice, etc.), eliminate the Service layer entirely. Controllers call Executors directly via a thin Facade. This removes 18 files and ~800 lines of boilerplate.

Option B — Elevate real orchestration: For Workflow and Task modules where cross-cutting concerns exist (transactions, multi-executor coordination), merge related Executors into the Service so it actually earns its keep. A deep Service should contain the business logic that's currently scattered across CmdExe/QryExe pairs.

Benefits

  • Locality: Business logic lives in one place instead of being split across Service + N Executors
  • Leverage: Callers deal with fewer modules; each remaining module has more depth
  • Testability: Fewer layers to mock; tests exercise real behavior through simpler interfaces
  • Bug reduction: Eliminates dead-parameter bug pattern (currentUserId not forwarded)
  • Cognitive load: Developers stop navigating 3+ files to understand one operation

Candidate #2: Workflow State Machine Scattered Across CmdExecutors

Strong

Files Involved

TaskApproveCmdExe.javaWorkflowSubmitCmdExe.java WorkflowCancelCmdExe.javaTaskRejectCmdExe.java TaskTerminateCmdExe.javaWorkflowResubmitCmdExe.java WorkflowDomainServiceImpl.javaProcessStatusEnum.java TaskStatusEnum.java

Problem

The workflow approval engine's state machine logic is leaked across multiple CmdExecutors. TaskApproveCmdExe.processAfterApprove() (line 59-82) contains the core state transition logic: determining next nodes, handling countersign completion, marking process as PASSED, creating tasks for next node.

WorkflowSubmitCmdExe.execute() independently implements state transitions: DRAFT→IN_PROGRESS, finding first node, creating tasks, recording submit action. These two executors (and others) each contain fragments of what should be a single, coherent state machine.

The WorkflowDomainServiceImpl was clearly intended to be this central orchestrator, but it remains a pass-through itself — delegating to the same fragmented CmdExeces/QryExecs.

Solution

Create a WorkflowStateMachine module (or deepen WorkflowDomainService) that:

  • Encapsulates all valid state transitions (DRAFT→IN_PROGRESS→PASSED/REJECTED/RETURNED/CANCELED)
  • Determines the next node after an action (the node-resolution algorithm currently in processAfterApprove)
  • Handles countersign semantics (all-original-tasks-approved check)
  • Provides a clean interface like transition(ProcessInstanceId, Action, ActorId) → ProcessState

CmdExecutors become thin: validate input → call stateMachine.transition() → done.

Benefits

  • Locality: All state transition rules in one place — adding a new state or transition means editing one file
  • Leverage: A small interface (transition()) hides complex multi-table orchestration
  • Testability: State machine can be unit-tested exhaustively without Spring context
  • Safety: Impossible to accidentally introduce invalid transitions (centralized validation)
  • Seam: The state machine interface is a natural seam — can swap implementations for testing

...and ~30 more identical patterns for getById, delete, simple save operations.

Problem

The codebase has 100+ Executor classes for ~18 domain entities. Many are trivially shallow: their entire implementation is mapper.selectX() + convertor.toDTO() — literally 3 lines of logic.

The deletion test: deleting MenuGetByIdQryExe and inlining its 3 lines into the caller would reduce total complexity (fewer files, fewer imports, fewer indirections). This module isn't earning its keep.

Each trivial Executor requires: a Java file, a @Component annotation, constructor injection, an interface-like execute() method, and a corresponding line in the ServiceImpl. That's ~15 lines of boilerplate for 3 lines of behavior.

Solution

Consolidate trivial CRUD executors into generic/reusable modules:

  • CrudGetByIdQryExe<T> — Generic executor that handles getById for any entity given a Mapper + Convertor
  • CrudPageQryExe<T> — Generic paginated query executor
  • CrudSaveCmdExe<T> — Generic insert executor with audit field population
  • CrudDeleteCmdExe<T> — Generic delete executor

Only create custom Executors when there's genuine business logic (validation, multi-step orchestration, complex queries). For example, MenuSaveCmdExe (with depth validation) and TaskApproveCmdExe (with state machine interaction) would remain custom.

Benefits

  • ~60-70 files eliminated for trivial CRUD operations
  • Leverage: One generic module serves 15+ entities instead of 60+ one-liners
  • Locality: Audit fields (createUserId, updateUserId) handled in one place, not forgotten in every CmdExe
  • Consistency: No more "this entity sets createTime, that one doesn't" bugs
  • Signal: A custom Executor now signals "there's real logic here" rather than noise

Candidate #4: User Role Resolution Logic Embedded in Query Executor

Worth Exploring

Files Involved

UserGetByIdQryExe.javaUserListByPageQryExe.java UserConvertor.javaUserDTO.java UserRoleDO.java

Problem

UserGetByIdQryExe (lines 37-87) contains 50 lines of role resolution logic embedded inside what should be a simple "get user by ID" query executor. It has 3 private methods:

  • convertToDTOsWithRoles() — converts users and enriches with roles
  • getUserRoleIdsMap() — batch-queries UserRoleDO table
  • getRoleIdToNameMap() — loads ALL active roles into memory on every call

This logic is almost certainly duplicated in UserListByPageQryExe (which also needs to return users with roles). The role resolution concern has zero locality — it's buried inside a specific QryExe.

Additionally, getRoleIdToNameMap() loads every active role from DB on every user fetch — an N+1-style performance issue hiding in plain sight.

Solution

Extract role resolution into a dedicated UserRoleEnricher module:

  • Interface: enrich(List<UserDTO> users) — adds roles/roleNames to each user DTO
  • Can be applied as a post-processing step by any query that returns UserDTOs
  • Can cache the roleId→name map (it changes rarely)
  • Becomes a reusable concern: any endpoint returning users gets consistent role data

Benefits

  • Locality: Role resolution logic exists in exactly one place
  • Eliminates duplication: UserGetByIdQryExe and UserListByPageQryExe share the same enrichment
  • Performance fix opportunity: Centralized caching of role name map
  • Testability: UserRoleEnricher can be tested independently with mock data
  • Seam: Interface allows swapping enrichment strategy (e.g., from DB to cache)

Candidate #5: Transaction Management Leakage & Audit Field Gaps

Speculative

Files Involved

TaskServiceImpl.javaWorkflowServiceImpl.java UserSaveCmdExe.java MenuSaveCmdExe.java DictSaveCmdExe.java MyMetaObjectHandler.java

Problem

Issue A — @Transactional on pass-through methods: TaskServiceImpl annotates every method with @Transactional (lines 44, 49, 56, 62, 68, 74, 80), but these methods just delegate to CmdExecutors. If the transaction boundary belongs to the business operation, it should be on the Executor that actually does the work, not on the pass-through Service method.

Issue B — Audit fields not populated: Per project spec (common-fields-spec.md), every DO must have createUserId/createdTime/updateUserId/updateTime. However, examining UserSaveCmdExe.execute() (line 20-32), MenuSaveCmdExe.execute() (line 16-36), and DictSaveCmdExe.execute() (line 17-30) — none of them set createUserId or updateUserId. The currentUserId parameter is accepted by the Service but never forwarded.

Audit field violation: The project coding standard mandates setting createUserId and updateUserId on create/update operations. The MyMetaObjectHandler auto-fills them from SecurityContext, but the explicit passing of currentUserId through Service→Executor is entirely dead code. Either remove the parameter or use it explicitly for audit trails.

Solution

For Issue A: Move @Transactional to the CmdExecutor that performs multi-step writes (e.g., TaskApproveCmdExe which writes to both task and approval_record tables). Remove from Service pass-through methods.

For Issue B: Decide on one approach: either rely fully on MyMetaObjectHandler (remove dead currentUserId parameters from all signatures) OR explicitly pass and set it in Executors. Mixing both approaches creates confusion about which code path is responsible.

Benefits

  • Clear transaction boundaries: @Transactional wraps actual work, not delegation
  • Audit correctness: Create/update user IDs are reliably recorded
  • Reduced confusion: No more "who sets the audit fields?" questions
  • Cleaner signatures: Removing dead parameters simplifies the entire call chain

Candidate #1 + #3 Combined: Flatten CRUD Modules

Why first? Candidates #1 (pass-through Services) and #3 (trivial Executor explosion) are two sides of the same coin. Together they represent the lowest-risk, highest-impact refactoring:

  • Impact: Eliminates ~70 files (~18 ServiceImpl + ~50 trivial Executors) and ~2000 lines of boilerplate
  • Risk: Low — pure structural change, no behavioral modifications needed
  • Side benefit: Fixes the dead currentUserId parameter bug across all CRUD operations
  • Signal improvement: Remaining custom Executors (like TaskApproveCmdExe, WorkflowSubmitCmdExe) stand out as "modules with real logic worth reading"
  • Paves the way: Once CRUD noise is removed, #2 (Workflow State Machine) becomes much easier to tackle because the interesting code is no longer buried among 100 trivial files
