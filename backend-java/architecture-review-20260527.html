架构审视报告

Zenith Admin System - 工作流管理系统

生成时间: 2026-05-27

项目概述

这是一个基于 Spring Boot 3.5 + MyBatis-Plus 的后台管理系统，采用分层架构：

  • project-api: 服务接口定义与DTO
  • project-model: 数据对象(DO)与Mapper
  • project-service: 服务实现
  • project-web: 控制器层
  • start: 启动模块

发现的架构问题

1. TaskService 与 WorkflowService 严重耦合

涉及文件: TaskServiceImpl.java, WorkflowServiceImpl.java

Strong

问题

两个服务类都直接依赖 8个Mapper，存在大量重复代码：

  • createTasksForNode() 方法在两个类中完全相同
  • getApprovers() 方法在两个类中完全相同
  • 审批记录创建逻辑重复
  • 流程状态转换逻辑分散

影响

违反DRY原则，修改工作流逻辑需要在多处同步修改，极易引入bug。服务类变成了"上帝类"，难以测试和维护。

解决方案

提取 WorkflowDomainService 领域服务，封装：

  • 审批人解析逻辑
  • 任务创建逻辑
  • 流程状态转换规则

2. 缺少工作流领域抽象

涉及文件: TaskServiceImpl.java, WorkflowServiceImpl.java

Strong

问题

工作流的核心领域概念没有抽象：

  • 流程实例(ProcessInstance) - 状态转换规则散落在多处
  • 任务(Task) - 任务生命周期管理分散
  • 审批节点(Node) - 节点类型处理逻辑分散
  • 审批人解析 - 三种审批人类型(角色/用户/上级)的解析逻辑重复

影响

这是典型的"贫血模型"问题。业务规则分散在服务方法中，难以形成清晰的领域边界。新增审批类型或流程类型需要修改多处代码。

解决方案

引入领域模型和领域服务：

  • ApproverResolver - 审批人解析策略接口
  • TaskLifecycle - 任务生命周期管理
  • ProcessStateMachine - 流程状态机

3. N+1 查询问题

涉及文件: TaskServiceImpl.java:convertToDTO, WorkflowServiceImpl.java:convertToDTO

Worth exploring

问题

convertToDTO 方法在循环中执行数据库查询：

// TaskServiceImpl.java:456-486
private TaskDTO convertToDTO(TaskDO dO) {
    // ...
    ProcessInstanceDO instance = processInstanceMapper.selectById(...); // N+1
    UserDO initiator = userMapper.selectById(...); // N+1
    ProcessTemplateDO template = processTemplateMapper.selectById(...); // N+1
    // ...
}

影响

分页查询10条任务可能触发 30+ 次数据库查询，严重影响性能。

解决方案

批量预加载关联数据，使用 Map 缓存后按需取用：

// 批量查询
List instanceIds = tasks.stream().map(TaskDO::getProcessInstanceId).toList();
Map instanceMap = processInstanceMapper.selectBatchIds(instanceIds)
    .stream().collect(Collectors.toMap(ProcessInstanceDO::getId, Function.identity()));

4. 测试与实现不匹配

涉及文件: UserServiceImplTest.java, UserServiceImpl.java

Worth exploring

问题

测试用例与实际服务实现签名不匹配：

  • 测试调用 userService.save(UserDTO)，但实际方法签名是 save(UserAddCmd, Long currentUserId)
  • 测试调用 userService.update(UserDTO)，但实际方法签名是 update(UserUpdateCmd, Long currentUserId)
  • 测试调用 userService.delete(Long)，但实际方法签名是 delete(Long, Long currentUserId)

影响

测试无法验证实际业务逻辑，可能存在未被发现的问题。服务依赖过多Mapper导致单元测试需要mock大量依赖。

解决方案

1. 更新测试用例以匹配当前API签名
2. 通过领域服务拆分降低mock复杂度
3. 考虑引入集成测试验证完整流程

5. 分页逻辑重复

涉及文件: 多个 *ServiceImpl.java

Speculative

问题

每个服务的分页方法都有相似的PageInfo构建代码：

PageInfo result = new PageInfo<>();
result.setTotal(pageInfo.getTotal());
result.setPageNum(pageInfo.getPageNum());
result.setPageSize(pageInfo.getPageSize());
result.setPages(pageInfo.getPages());
result.setList(dtos);
return result;

解决方案

已有 PageResponseUtils 工具类，但仅在Controller层使用。可扩展为通用的分页转换工具。

6. 异常处理不规范

涉及文件: 所有 *ServiceImpl.java

Speculative

问题

所有异常都使用 RuntimeException：

throw new RuntimeException("任务不存在");
throw new RuntimeException("流程实例不存在");
throw new RuntimeException("当前状态不允许取消");

影响

无法区分业务异常类型，前端难以做针对性处理。日志分析困难。

解决方案

引入业务异常体系：BusinessException 基类 + 具体异常类型，配合 GlobalExceptionHandler 统一处理。

优先建议

首选重构目标：提取 WorkflowDomainService

这是影响最大的重构，能同时解决代码重复、测试困难和领域抽象缺失的问题。

消除代码重复 提升测试性 建立领域边界 降低服务复杂度
