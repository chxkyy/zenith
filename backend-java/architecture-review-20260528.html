🔴 Service 层直接操作 DO 对象

Strong

📁 涉及文件

WorkflowServiceImpl.java (381行) TaskServiceImpl.java (363行) UserServiceImpl.java (267行) DictServiceImpl.java (188行) RoleServiceImpl.java (149行) OrgServiceImpl.java (114行)

❌ 问题描述

根据 COLA 架构规范，Service 层应该只做用例编排，将业务逻辑下沉到 Executor 层。 但当前代码中，所有 Service 实现类都直接注入 Mapper 并操作 DO 对象，严重违反分层原则。

💻 违规代码示例

// WorkflowServiceImpl.java - Service 层直接操作 DO
@Service
@RequiredArgsConstructor
public class WorkflowServiceImpl implements WorkflowService {
    private final ProcessInstanceMapper processInstanceMapper;  // ❌ 直接注入 Mapper
    private final TaskMapper taskMapper;                         // ❌ 直接注入 Mapper
    private final UserMapper userMapper;                         // ❌ 直接注入 Mapper
    
    public void cancel(Long processInstanceId, Long currentUserId) {
        ProcessInstanceDO instance = processInstanceMapper.selectById(processInstanceId);  // ❌ 直接操作 DO
        instance.setStatus(ProcessStatusEnum.CANCELLED.getCode());
        processInstanceMapper.updateById(instance);  // ❌ 直接操作 DO
    }
}

✅ 解决方案

为每个 Service 创建对应的 Executor 类，将 DO 操作下沉到 Executor 层。 Service 层只负责用例编排，调用 Executor 执行具体业务逻辑。

🎯 收益

  • • Locality（局部性）：业务逻辑集中在 Executor，修改影响范围小
  • • Leverage（杠杆）：Service 接口简洁，Executor 可复用
  • • Testability（可测试性）：Executor 可独立单元测试
  • • AI-Navigability：清晰的分层使 AI 更容易理解和导航代码

🔴 AuthService 返回 COLA Response 对象

Strong

📁 涉及文件

AuthServiceImpl.java AuthService.java (接口)

❌ 问题描述

Service 层返回了 COLA Response 对象（SingleResponse、Response），违反了分层原则。 这导致 AOP 切面（TranslateAspect）无法正确拦截和处理返回值，字段翻译功能失效。

💻 违规代码示例

// AuthServiceImpl.java - Service 层返回 COLA Response
@Service
@RequiredArgsConstructor
public class AuthServiceImpl implements AuthService {
    @Override
    public SingleResponse<UserDTO> login(String loginId, String password, String ip, String userAgent) {
        UserDO user = userMapper.selectOne(...);
        if (user == null) {
            return SingleResponse.buildFailure("USER_NOT_FOUND", "登录账号不存在");  // ❌ 返回 COLA Response
        }
        UserDTO userDTO = userService.getById(user.getId());
        return SingleResponse.of(userDTO);  // ❌ 返回 COLA Response
    }
    
    @Override
    public Response changePassword(Long userId, String oldPassword, String newPassword) {
        // ...
        return Response.buildSuccess();  // ❌ 返回 COLA Response
    }
}

✅ 解决方案

Service 层应该返回原始数据类型或抛出异常，由 Controller 层负责包装成 COLA Response。

📈 架构对比

⚠️ 后果

  • • AOP 切面（字段翻译）无法正确拦截
  • • @UserName / @RoleName / @DictName 等注解失效
  • • 违反架构分层原则，Service 层承担了 Controller 的职责

⚪ AOP 切面与 COLA Response 冲突

Speculative

❌ 问题描述

TranslateAspect 只处理 List、PageInfo 和单个 DTO 对象，不处理 COLA Response 对象。 当 Service 返回 SingleResponse 时，AOP 无法正确执行字段翻译。

💻 当前 AOP 实现

// TranslateAspect.java
@AfterReturning(pointcut = "servicePointCut()", returning = "result")
public void afterReturn(Object result) {
    if (result == null) return;
    
    // 只处理这三种类型，不处理 COLA Response
    if (result instanceof PageInfo<?>) {
        PageInfo<?> pageInfo = (PageInfo<?>) result;
        pageInfo.getList().forEach(fieldTranslateProcessor::process);
    } else if (result instanceof List<?>) {
        List<?> list = (List<?>) result;
        list.forEach(fieldTranslateProcessor::process);
    } else {
        fieldTranslateProcessor.process(result);
    }
    // ❌ 如果 result 是 SingleResponse，会尝试对 Response 对象做字段翻译，而不是对内部 data
}

💡 根本原因

这个问题的根本原因是 Service 层返回了 COLA Response 对象。 如果按照规范让 Service 返回原始数据类型，AOP 就能正确工作。

🎯 首要推荐

优先修复：AuthServiceImpl 返回 COLA Response

这是影响范围最小、修复成本最低的问题。只需修改 AuthService 接口和实现， 让 Service 层返回原始数据类型，由 Controller 层负责包装响应。
